# Философия работы над проектом

## Принципы

### 1. Сначала — разберись, потом — пиши
Перед тем как вносить изменения, изучи связанные файлы, imports и зависимости. Не создавай несуществующие переменные, функции или компоненты.

### 2. Один шаг за раз
Не пытайся переписать всё сразу. Вноси изменения итеративно — один компонент, одна фича, один баг за раз. Проверяй результат после каждого шага.

### 3. Макет Figma — источник правды
Все визуальные решения сверяются с макетом Figma. При расхождении между кодом и макетом — приоритет у макета.

### 4. Не дублируй — редактируй
Никаких файлов с `_fix`, `_v2`, `_final`. Если нужно исправить компонент — редактируй оригинальный файл. Git хранит историю.

### 5. Сохраняй контекст
После каждой рабочей сессии обновляй `CURRENT_STAGE.md`. Это экономит время и токены при следующем запуске.

## Перед каждым изменением — чеклист

1. Прочитал ли я файл, который собираюсь менять?
2. Проверил ли я imports и exports связанных модулей?
3. Не сломаю ли я другие компоненты этим изменением?
4. Соответствует ли изменение макету Figma?
5. Использую ли я существующие UI-компоненты из `src/app/components/ui/`?

## Порядок работы над фичей

1. Сверить макет Figma с текущим состоянием компонента
2. Определить разницу — что нужно изменить
3. Проверить зависимости (imports, props, стили)
4. Внести изменения
5. Проверить результат в браузере
6. Обновить `CURRENT_STAGE.md`
